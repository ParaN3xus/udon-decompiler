# Udon Decompiler: LLM Context

## Project summary

- Python decompiler for UdonSharp/Udon programs.
- Input: dumped Udon program JSON generated by Unity editor scripts.
- Output: pseudo C# source formatted by `clang-format`.

## High-level flow (current implementation)

1. Unity editor scripts dump `serializedProgramCompressedBytes` into JSON (`Editor/UdonProgramDumper.cs`).
2. `ProgramLoader` parses JSON into `UdonProgramData`.
3. `BytecodeParser` decodes bytecode into `Instruction` objects (big-endian u32 opcodes/operands).
4. `CFGBuilder` builds per-function CFGs and discovers hidden/internal function entry points.
5. `FunctionDataFlowAnalyzer` runs stack simulation + variable identification + raw IR build (`IRBuilder`).
6. `TransformPipeline` runs IL transforms and program transforms to lift/simplify control flow and build `IRClass`.
7. `ProgramCodeGenerator` invokes `CSharpCodeGenerator` to emit and format C#.

## Key entry points

- CLI: `udon-decompiler` (`udon_decompiler/__main__.py`)
- Library API: `decompile_program_to_source` (`udon_decompiler/__init__.py`)

## CLI behavior notes

- `--info` defaults to `./UdonModuleInfo.json` and is required at runtime.
- Input can be a single `.json` file or a directory of `.json` files.
- For directory input, output defaults to `<input-dir>-decompiled`.
- `UdonModuleInfo.json` in an input directory is skipped from decompilation.

## Dumped program JSON format (input)

`ProgramLoader` expects:

- `byteCodeHex`: hex string of bytecode
- `byteCodeLength`: bytecode length in bytes
- `symbols`: map `name -> {name, type, address}`
- `entryPoints`: list of `{name, address}`
- `heapInitialValues`: map `address(string) -> {address, type, value:{isSerializable, value}}`

Mapping details:

- `heapInitialValues` keys are parsed to `int`.
- `entryPoints[i].address` initializes both `EntryPointInfo.address` and `call_jump_target`.
- Later, `BytecodeParser` adjusts `call_jump_target` for function headers with `PUSH __const_SystemUInt32_0` (halt sentinel): `call_jump_target = address + OpCode.PUSH.size`.

## Repo layout (core)

- `udon_decompiler/loaders/`: `ProgramLoader`, `ModuleInfoLoader`

- `tests/`: markdown cases, case tooling, snapshot and unit tests

## Core data structures

- `UdonProgramData` (`models/program.py`)
  - stores bytecode/symbols/entry points/heap values
  - `get_class_name()` checks heap address `1` (`__refl_typename`)
- `Instruction` and `OpCode` (`models/instruction.py`)
- `UdonModuleInfo` (`models/module_info.py`) global extern signature registry

## Disassembly and CFG/dataflow analysis

- `BytecodeParser` parses sequentially and annotates operands (`operand_name` for symbol-backed operands/extern signatures).
- `CFGBuilder`:
  - identifies basic blocks (`BasicBlockIdentifier`)
  - detects switch-table `JUMP_INDIRECT` patterns
  - detects internal/hidden entries from call-like jumps
  - builds `ControlFlowGraph` per function
- `FunctionDataFlowAnalyzer` per function:
  - `_simulate_stack()` using `StackSimulator`
  - `_identify_variables()` using `VariableIdentifier`
  - `_build_ir()` via `IRBuilder`

## Transform pipeline (no legacy SCFG path)

Current default IL transform order (`build_default_pipeline()`):

1. `ControlFlowSimplification`
2. `ConstToLiteral`
3. `TempVariableInline`
4. `DetectExitPoints(can_introduce_exit_for_return=False)`
5. `BlockILTransform(post_order=[LoopDetection])`
6. `DetectExitPoints(can_introduce_exit_for_return=True)`
7. `BlockILTransform(post_order=[ConditionDetection])`
8. `HighLevelLoopTransform`
9. `HighLevelSwitchTransform`
10. `HighLevelLoopStatementTransform`
11. `StructuredControlFlowCleanupTransform`
12. `CollectLabelUsage`
13. `CollectVariables`

Program-level transforms:

1. `IRClassConstructionTransform`
2. `PromoteGlobals`

Notes:

- `IRClassConstructionTransform` creates `IRClass` (class/namespace/functions).
- `PromoteGlobals` moves globals/shared locals to class-level declarations.
- Old `numba-scfg`/`SCFGAdapter`/`ASTBuilder` lifter flow is not part of current code.

## Code generation

- `ProgramCodeGenerator.generate_program()` takes `IRClass` and returns `(class_name, code)`.
- `CSharpCodeGenerator` emits pseudo C# from IR statements/expressions.
- It supports structured nodes (`IRHighLevelWhile`, `IRHighLevelDoWhile`, `IRHighLevelSwitch`) plus fallback unstructured flow (`goto` labels).
- Final formatting is done through `clang-format` (`clang_format.get_executable("clang-format")`).

## Tests and snapshots

- Cases: `tests/cases/**/*.md` (excluding `README.md`)
- Markdown parser requires exactly 1 or 2 fenced code blocks (`tests/ci/md_cases.py`)
  - block 1: UdonSharp C# source
  - block 2: dumped JSON (if missing/empty, snapshot test skips that case)
- Snapshot test: `tests/test_snapshots.py`
  - loads dumped JSON block
  - decompiles
  - compares with `tests/cases/**/snaps/*.cs` via `syrupy`

Useful flows:

1. Edit case markdown.
2. Let `compile-test-cases.yml` regenerate dumped JSON blocks.
3. Run `uv run pytest --snapshot-update`.
4. Review and commit snapshot changes.

Single-case debug helper:

- `python -m tests.decompile_case tests/cases/control_flow/SwitchCaseLong.md`
- It prints both GT source and decompiled source.
- Add `--debug-log` to enable DEBUG logs during decompilation.

## Udon quick primer

- Udon is a stack VM used by VRChat; UdonSharp compiles to Udon bytecode.
- Stack values are mostly heap addresses, not immediate language-level values.
- Key opcodes used here: `NOP`, `PUSH`, `POP`, `COPY`, `JUMP`, `JUMP_IF_FALSE`, `JUMP_INDIRECT`, `EXTERN`, `ANNOTATION`.
- `EXTERN` operand resolves to a signature string in heap, then to function metadata in `UdonModuleInfo`.
